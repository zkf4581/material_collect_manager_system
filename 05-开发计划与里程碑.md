# 开发计划与里程碑

## 1. 计划原则

- 优先打通业务闭环
- 优先让多人可并行开发
- 优先完成可演示、可试用的最小版本
- 每一阶段都要有可验收交付物

## 2. 阶段划分

## 阶段 0：项目启动与基础准备

时间建议：`2~3 天`

交付物：

- 项目文档
- 仓库目录骨架
- 前后端脚手架
- 数据库初始化脚本
- 接口规范约定

验收标准：

- 本地能启动前端、后台、后端
- 数据库可初始化
- 团队可开始并行开发

## 阶段 1：基础数据与登录

时间建议：`3~5 天`

交付物：

- 登录鉴权
- 项目、班组、工人基础管理
- 材料分类与积分规则管理

验收标准：

- 后台能维护基础主数据
- H5 与后台都能登录
- 前后端路由与权限跑通

## 阶段 2：回收登记闭环

时间建议：`5~7 天`

交付物：

- H5 回收登记页
- 图片上传
- 回收记录列表
- 后台回收记录审核
- 自动积分计算与积分流水

验收标准：

- 现场可提交一条完整回收记录
- 后台可审核
- 审核通过后工人积分变化正确

## 阶段 3：积分查询与兑换

时间建议：`4~6 天`

交付物：

- H5 我的积分页
- H5 积分流水页
- 商品管理
- 兑换申请
- 兑换审核与扣分

验收标准：

- 工人可查看积分和流水
- 可提交兑换
- 审核通过后积分正确扣减

## 阶段 4：报表、优化与上线准备

时间建议：`3~5 天`

交付物：

- 基础统计报表
- 表单优化
- 错误处理与空状态处理
- 本地部署文档
- 上云部署预案

验收标准：

- 主要角色可顺畅使用
- 关键报表可查看
- 具备部署到测试环境的条件

## 3. 推荐排期

整体建议周期：`3~5 周`

```text
第 1 周：启动准备 + 基础数据 + 登录
第 2 周：回收登记 + 图片上传 + 积分计算
第 3 周：积分查询 + 积分兑换
第 4 周：审核完善 + 报表 + 联调测试
第 5 周：缓冲与试运行修复
```

## 4. 并行开发建议

### 4.1 第一波并行

- 前端 H5 基础骨架
- 后台基础骨架
- 服务端基础骨架
- 数据库脚本

### 4.2 第二波并行

- H5 回收登记
- 服务端回收记录与上传
- 后台基础数据管理

### 4.3 第三波并行

- H5 积分查询
- 服务端积分账户与流水
- 后台回收审核

### 4.4 第四波并行

- H5 兑换页面
- 服务端兑换申请与扣分
- 后台兑换审核与商品管理

## 5. 关键风险与应对

### 5.1 业务风险

- 积分规则频繁调整
- 审核流程不明确
- 基础字典迟迟定不下来

应对：

- 第一版只支持简单积分规则
- 所有规则配置化
- 提前冻结第一版字段和流程

### 5.2 技术风险

- 图片上传兼容性问题
- 权限模型过度设计
- 多人修改同一模块产生冲突

应对：

- 先在主流手机浏览器验证上传
- 权限模型先做最小可用版
- 严格模块分工与分支管理

## 6. 每阶段测试要求

- 单元测试：服务端核心业务逻辑
- 接口测试：登录、上传、回收、兑换
- 页面测试：H5 提交流程、后台审核流程
- 回归测试：积分增减与状态流转

## 7. 当前建议的首批交付顺序

1. 先建项目目录与脚手架
2. 先落数据库脚本与基础实体
3. 再做后台基础数据管理
4. 再做 H5 回收登记
5. 最后补积分兑换和报表
