# 接口与状态机约定

## 1. 文档目标

本文件用于在多人并行开发前冻结关键契约，降低前后端联调成本，避免不同 Agent 各自理解不同导致返工。

## 2. 统一接口约定

### 2.1 基础规范

- 接口统一前缀：`/api`
- 数据格式：`application/json`
- 文件上传：`multipart/form-data`
- 时间格式：`yyyy-MM-dd HH:mm:ss`
- 分页参数：`pageNum`、`pageSize`

### 2.2 统一返回结构

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "traceId": "8f2b4f2d2f5d4d8a"
}
```

返回字段说明：

- `code`：业务状态码，`0` 表示成功
- `message`：提示信息
- `data`：业务数据
- `traceId`：链路追踪 ID，便于排查问题

### 2.3 分页返回结构

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "list": [],
    "pageNum": 1,
    "pageSize": 10,
    "total": 0
  },
  "traceId": "8f2b4f2d2f5d4d8a"
}
```

## 3. 统一错误码建议

- `0`：成功
- `40001`：参数错误
- `40002`：文件类型不支持
- `40003`：文件大小超限
- `40101`：未登录或 Token 失效
- `40301`：无权限访问
- `40401`：资源不存在
- `40901`：状态冲突
- `40902`：积分不足
- `40903`：库存不足
- `50001`：系统异常

## 4. 鉴权约定

### 4.1 登录方式

第一版建议先使用：

- 后台：账号密码登录
- H5：账号密码登录，后续可扩展手机号验证码

### 4.2 Token 传递

前端通过请求头传递：

```text
Authorization: Bearer <token>
```

### 4.3 数据权限

- 工人只能查看自己的积分、记录、兑换
- 库管只能提交和查看本项目数据
- 项目管理员只能查看本项目范围数据
- 公司管理员可查看全部项目

## 5. 核心状态机

## 5.1 回收记录状态

状态值建议：

- `DRAFT`：草稿
- `SUBMITTED`：已提交待审核
- `APPROVED`：已审核通过
- `REJECTED`：已驳回
- `CANCELLED`：已作废

状态流转：

```text
DRAFT -> SUBMITTED
SUBMITTED -> APPROVED
SUBMITTED -> REJECTED
APPROVED -> CANCELLED
REJECTED -> DRAFT
```

流转约束：

- `APPROVED` 后才能正式发放积分
- `CANCELLED` 必须有作废原因
- 作废已通过记录时必须产生积分冲销流水

## 5.2 兑换申请状态

状态值建议：

- `SUBMITTED`：待审核
- `APPROVED`：审核通过待发放
- `REJECTED`：已驳回
- `DELIVERED`：已发放
- `CANCELLED`：已取消

状态流转：

```text
SUBMITTED -> APPROVED
SUBMITTED -> REJECTED
APPROVED -> DELIVERED
SUBMITTED -> CANCELLED
```

流转约束：

- `APPROVED` 前必须校验积分余额和库存
- `DELIVERED` 时必须记录发放人和发放时间
- 驳回必须填写原因

## 6. 核心接口清单

## 6.1 认证

- `POST /api/auth/login`
- `GET /api/auth/me`
- `POST /api/auth/logout`

## 6.2 基础数据

- `GET /api/projects`
- `POST /api/projects`
- `GET /api/teams`
- `POST /api/teams`
- `GET /api/workers`
- `POST /api/workers`
- `GET /api/material-items`
- `POST /api/material-items`
- `GET /api/point-rules`
- `POST /api/point-rules`

## 6.3 文件上传

- `POST /api/files/upload`

上传返回示例：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "fileId": 1001,
    "url": "/uploads/2026/03/22/1/3f4e8b.jpg",
    "storageType": "LOCAL",
    "size": 234567
  },
  "traceId": "8f2b4f2d2f5d4d8a"
}
```

## 6.4 回收登记

- `POST /api/recycle-records`
- `GET /api/recycle-records`
- `GET /api/recycle-records/{id}`
- `POST /api/recycle-records/{id}/submit`
- `POST /api/recycle-records/{id}/approve`
- `POST /api/recycle-records/{id}/reject`
- `POST /api/recycle-records/{id}/cancel`

创建回收记录请求示例：

```json
{
  "projectId": 1,
  "teamId": 11,
  "workerId": 100,
  "materialItemId": 2001,
  "quantity": 35,
  "unitCode": "KG",
  "conditionCode": "OK",
  "photoIds": [1001, 1002],
  "remark": "钢筋余料回收"
}
```

## 6.5 积分

- `GET /api/points/accounts`
- `GET /api/points/ledger`

## 6.6 兑换

- `GET /api/reward-items`
- `POST /api/reward-items`
- `POST /api/exchange-orders`
- `GET /api/exchange-orders`
- `POST /api/exchange-orders/{id}/approve`
- `POST /api/exchange-orders/{id}/reject`
- `POST /api/exchange-orders/{id}/deliver`

## 7. 关键数据契约

## 7.1 回收记录积分快照

回收记录中建议保存 `pointCalcSnapshot`，示例：

```json
{
  "ruleId": 12,
  "materialItemId": 2001,
  "unitCode": "KG",
  "basePoint": 1.0,
  "conditionCode": "OK",
  "conditionFactor": 1.0,
  "quantity": 35,
  "finalPoints": 35
}
```

这样即使后面规则变了，也能解释历史积分。

## 7.2 积分流水业务类型

- `RECYCLE_AWARD`：回收奖励
- `RECYCLE_REVERSE`：回收冲销
- `EXCHANGE_DEDUCT`：兑换扣减
- `MANUAL_ADJUST`：人工调整

## 8. 开发冻结建议

进入多人并行开发前，以下内容必须先冻结：

1. 角色与数据权限范围
2. 回收记录状态机
3. 兑换申请状态机
4. 核心数据库表主字段
5. 统一返回结构与错误码
6. 上传接口返回格式

## 9. 联调前检查清单

1. H5 与后台是否都使用同一套错误码
2. 上传接口返回结构是否统一
3. 回收记录提交后是否一定产生状态流转
4. 审核通过后是否一定产生积分流水
5. 兑换通过后是否一定扣减积分并更新库存
